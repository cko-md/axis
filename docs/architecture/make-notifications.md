# Make notification boundary

## Ownership

Axis owns deterministic detection, recipient selection, the notification
payload, and an append-only audit trail. Make owns the configured scenario and
the final email delivery. A `2xx` response means Make accepted the webhook; it
does not prove that the downstream email provider delivered the message.

The webhook URLs are server-only environment values. They contain opaque
credentials and must never appear in logs, Sentry, responses, or persisted
audit metadata.

## Event map

| Axis event | Trigger | Make URL environment variable | Idempotency key |
|---|---|---|---|
| `daily_brief` | Daily finance job writes the current deterministic brief | `MAKE_WEBHOOK_DAILY_BRIEF_URL` | user + snapshot date |
| `weekly_recap` | Weekly recap is due and has been persisted | `MAKE_WEBHOOK_WEEKLY_RECAP_URL` | user + snapshot date |
| `bill_reminder` | Active recurring bill is due in two days | `MAKE_WEBHOOK_BILL_REMINDER_URL` | user + merchant + due date |
| `budget_alert` | Category spend reaches at least 90% of its monthly limit | `MAKE_WEBHOOK_BUDGET_ALERT_URL` | user + category + month |
| `anomaly_alert` | Deterministic Activity rules flag a transaction | `MAKE_WEBHOOK_ANOMALY_ALERT_URL` | user + transaction id |
| `subscription_audit` | Weekly stale-subscription review is due | `MAKE_WEBHOOK_SUBSCRIPTION_AUDIT_URL` | user + review date |

All six events are external communications generated by the owner-configured
finance automation. They never authorize or execute a financial transaction.
Authoritative monetary values are computed by Axis before the payload reaches
Make; Make must not alter or infer them.

## Delivery contract

`src/lib/integrations/make.ts` is the outbound adapter. It:

- accepts only HTTPS `hook.<zone>.make.com` URLs with no credentials or port;
- disables redirects so an opaque hook cannot redirect a server-side request;
- performs one attempt only because blind write retries can duplicate email;
- treats only `2xx` as accepted and returns the shared structured `Result`;
- emits safe provider failure metadata without URL paths, response bodies, or
  payload content.

`src/lib/fund/notifyViaMake.ts` writes an immutable
`pending_confirmation` audit row before calling Make and an immutable
`success` or `failure` row afterward. A failed preflight audit blocks delivery.
Failure rows contain only the idempotency key, normalized error code, HTTP
status, and retryability flag.

The idempotency key is included in every webhook payload. Each Make scenario
must reject a key it has already processed before invoking its email module.
Axis does not automatically retry these writes. A future durable outbox can
add operator-controlled replay and dead-letter handling without weakening the
external-communication boundary.

## Runtime verification

For each configured scenario:

1. Send one non-production test payload and confirm Make returns `2xx`.
2. Confirm the scenario rejects the same idempotency key on a second request.
3. Confirm `audit_logs` contains pending plus success/failure rows and no
   recipient, subject, body, webhook URL, or provider response body.
4. Query Sentry for `provider=make`, `operation=trigger_webhook`; verify a forced
   upstream failure records only normalized code/status metadata.
